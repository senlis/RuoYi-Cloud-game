# 游戏运营管理平台 — 项目/渠道/分区/服务器 需求设计文档

> 基于 RuoYi-Cloud v3.6.8，功能放在 ruoyi-modules-system 模块中
> 前端：Vue 2 + Element UI（后续升级 Vue 3）

---

## 一、核心实体关系

```
游戏项目 (Game Project)
  └── 渠道 (Channel)                    ← 一个项目下多个渠道
        └── 分区 (Region/Partition)      ← 一个渠道下多个大区
              └── 服务器 (Server)         ← 一个大区下多个游戏服
```

**业务场景：**
- **项目**：一款游戏就是一个项目，比如《梦幻西游》《大话西游》。不同项目独立运营
- **渠道**：推广分发渠道，如九游、华为、应用宝、官网等
- **分区**：按平台或运营策略划分的大区，如 iOS 大区、Android 大区、混服大区
- **服务器**：具体的游戏服务器，如 iOS 1服、2服...

## 二、动态字段系统设计 ⭐

这是整个模块的核心扩展能力。四个实体都支持动态字段，新增字段无需改代码。

### 2.1 设计思路

```
定义层（运行时增删改）                 数据层（每个实体记录）
┌─────────────────────┐          ┌──────────────────────┐
│ game_field_define   │   指引    │ 各实体表 +           │
│ entity_type         │───────→  │ dynamic_fields(JSON) │
│ field_label/code    │          │ 字段值存储于此        │
│ field_type/is_export│          └──────────────────────┘
│ is_required/sort    │          ┏━━━━━━━━━━━━━━━━━━━━┓
│ status              │          ┃ 查询/展示           ┃
└─────────────────────┘          ┃ 从 define 读取元数据┃
                                 ┃ 从 dynamic_fields   ┃
                                 ┃ 读取具体值          ┃
                                 ┗━━━━━━━━━━━━━━━━━━━━┛
```

### 2.2 动态字段定义表 `game_field_define`

```sql
CREATE TABLE game_field_define (
  field_id      bigint(20)   NOT NULL AUTO_INCREMENT  COMMENT '字段ID',
  entity_type   varchar(50)  NOT NULL  COMMENT '所属实体(project/channel/region/server)',
  field_label   varchar(100) NOT NULL  COMMENT '字段显示名称(如"游戏简称")',
  field_code    varchar(100) NOT NULL  COMMENT '字段编码(如 game_abbr)，唯一',
  field_type    varchar(50)  NOT NULL  COMMENT '字段类型: text/number/date/select/boolean',
  field_options text         COMMENT '选项配置(select类型用，JSON数组 如[{"label":"A","value":"a"}])',
  is_export     char(1)      DEFAULT 'N'  COMMENT '是否导出(Y/N)，导出配置时使用',
  is_required   char(1)      DEFAULT 'N'  COMMENT '是否必填',
  default_value varchar(500) COMMENT '默认值',
  sort          int(4)       DEFAULT 0  COMMENT '排序号',
  status        char(1)      DEFAULT '0' COMMENT '状态(0正常 1停用)',
  remark        varchar(500) COMMENT '备注',
  create_by     varchar(64),
  create_time   datetime,
  update_by     varchar(64),
  update_time   datetime,
  PRIMARY KEY (field_id),
  UNIQUE KEY uk_entity_field (entity_type, field_code)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='动态字段定义表';
```

### 2.3 字段存储方式

每个实体表的 `dynamic_fields` 列（TEXT 类型）存储 JSON：

```json
{
  "game_abbr": "MHXY",
  "publisher": "网易",
  "custom_rating": "4.5"
}
```

**NULL** 表示该记录没有动态字段数据。

### 2.4 字段类型支持

| field_type | 前端组件 | 说明 |
|------------|----------|------|
| text | `<el-input>` | 单行文本 |
| number | `<el-input-number>` | 数字 |
| textarea | `<el-input type="textarea">` | 多行文本 |
| date | `<el-date-picker>` | 日期选择 |
| select | `<el-select>` | 下拉选择，options 定义选项 |
| boolean | `<el-switch>` | 开关，存储 true/false |

### 2.5 动态字段管理页面

提供一个统一的「动态字段配置」页，用于管理所有实体的动态字段：

- **页面位置**：系统管理 → 动态字段配置
- **功能**：按实体类型（项目/渠道/分区/服务器）管理字段
- **操作**：新增字段、编辑字段、调整排序、启用/停用、删除
- **表单内容**：所属实体、字段名称、字段编码、字段类型、选项配置、是否导出、是否必填、排序号

> 权限标识：`game:field:list` `game:field:add` `game:field:edit` `game:field:remove`

## 三、数据库表设计

### 3.1 游戏项目表 `game_project`

```sql

### 3.3 游戏分区表 `game_region`

  export_config text         COMMENT '导出配置(JSON)，生成后存储在此',

  update_by     varchar(64),

  KEY idx_channel (channel_id),
  KEY idx_project (project_id)

```

### 3.4 游戏服务器表 `game_server`

```sql
CREATE TABLE game_server (
  server_id       bigint(20)   NOT NULL AUTO_INCREMENT  COMMENT '服务器ID',
  region_id       bigint(20)   NOT NULL  COMMENT '所属分区ID',
  channel_id      bigint(20)   NOT NULL  COMMENT '所属渠道ID(冗余)',
  project_id      bigint(20)   NOT NULL  COMMENT '所属项目ID(冗余)',
  server_code     varchar(50)  NOT NULL  COMMENT '服务器编码(如 s1、s2)',
  server_name     varchar(100) NOT NULL  COMMENT '服务器名称(如"iOS 1服")',
  server_type     char(1)      DEFAULT '0' COMMENT '服务器类型(0=正式 1=测试 2=体验)',
  status          char(1)      DEFAULT '0' COMMENT '状态(0=正常 1=维护 2=停服)',
  open_time       datetime     COMMENT '开服时间',
  backend_url     varchar(255) COMMENT '后台交互地址(游戏服HTTP API地址)',
  game_db_config  text         COMMENT '游戏数据库配置(JSON)',
  log_db_config   text         COMMENT '游戏日志数据库配置(JSON)',
  merge_info      text         COMMENT '合服信息(JSON数组)',
  sort            int(4)       DEFAULT 0  COMMENT '显示顺序',
  dynamic_fields  text         COMMENT '动态字段值(JSON)',
  remark          varchar(500) COMMENT '备注',
  create_by       varchar(64),
  create_time     datetime,
  update_by       varchar(64),
  update_time     datetime,
  PRIMARY KEY (server_id),
  KEY idx_region (region_id),
  KEY idx_channel (channel_id),
  KEY idx_project (project_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='游戏服务器表';
```

## 六、分区导出配置功能详解 ⭐

### 6.1 功能描述

点击分区的「导出配置」按钮，将区服的固定字段 + 标记为可导出的动态字段 + 关联的服务器信息 统一生成 JSON 配置，存储到数据库并更新缓存。

> 注意：`backend_url`、`game_db_config`、`log_db_config` 属于内部运维配置，**不导出**到客户端 JSON。

### 6.2 生成的 JSON 结构

```json
{
  "region_id": 1,
  "region_code": "ios_1",
  "region_name": "iOS官方大区",
  "status": "0",
  "sort": 1,
  "region_recommend": "1",
  "region_tag": "最新",
  "servers": [
    {
      "server_id": 1,
      "server_code": "s1",
      "server_name": "iOS 1服",
      "server_type": "0",
      "status": "0",
      "open_time": "2025-01-01 10:00:00",
      "server_load": "良好",
      "is_new": "true"
    }
  ]
}
```

### 6.3 数据流

```
用户点击「导出配置」
  → 后端读取 region 固定字段 + 可导出动态字段
  → 读取该分区下所有 servers（排除 backend_url, game_db_config, log_db_config）
  → 读取 servers 的可导出动态字段
  → 组装 JSON
  → 存储到 game_region.export_config 字段
  → 更新 Redis 缓存（key: game:region:export:{regionId}）
  → 返回成功

客户端获取配置（无需认证）：
  GET /game/region/{regionId}/clientConfig
  → 优先从 Redis 读取
  → 若缓存过期则从 DB 读取 export_config 字段
  → 返回 JSON
```

## 七、分区克隆配置功能

### 7.1 功能描述

将某个分区（包含其关联服务器的配置）克隆为模板，快速创建新分区。

### 7.2 克隆逻辑

```
1. 读取源分区的固定字段 + 动态字段值
2. 创建新分区记录（新名称 + 新编码）
3. 如果选择了「克隆服务器配置」：
   - 复制源分区的所有服务器（不复制 server_code/name）
   - 复制服务器的固定配置 + 动态字段值
4. 返回新分区 ID
```

## 十一、联动与校验规则

1. **四级联动筛选**：服务器列表支持 项目→渠道→分区→服务器名 多级联动筛选
2. **级联删除保护**：删除项目时检查是否有渠道；删除渠道时检查是否有分区；删除分区时检查是否有服务器
3. **字段编码唯一性**：动态字段的 `field_code` 在同一实体类型下唯一
4. **导出配置更新**：分区下服务器发生变化时，已有的导出配置标记为"需更新"
5. **缓存策略**：客户端配置缓存到 Redis，TTL 5 分钟
6. **后端地址格式校验**：`backend_url` URL 格式校验
7. **数据库配置校验**：`game_db_config`、`log_db_config` JSON 语法校验
